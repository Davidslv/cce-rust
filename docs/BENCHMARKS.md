# Benchmarks (v2 — language packs)

Generated by `cce bench --lang <language>` using the default deterministic
**hash** embedder. The four languages in active use (SPEC-V2 §8) are benchmarked
against a pinned real repository each; Python and JavaScript remain validated
packs but ship no labeled benchmark corpus.

## Environment

| Field | Value |
|---|---|
| Engine | Rust (`rustc 1.91.1`) |
| OS / Arch | macOS / arm64 |
| Embedder | hash (SPEC §5.1) |
| Date | 2026-07-05 |

## Interpretation

With the hashing embedder, retrieval is essentially lexical (SPEC §10 note), so
recall reflects keyword overlap between the query and code identifiers rather
than semantic similarity — yet across four very different languages and code
styles the labeled query sets land a correct file in the top-10 for 80–90% of
queries, a good result for a model-free baseline. Token savings are large
everywhere (62–96%) because a query returns a handful of function- or
struct-sized chunks instead of the whole files they live in — the entire point
of the engine: feed a model precise snippets, not entire files. C saves the most
because jq's translation units are large relative to the individual functions the
chunker isolates; Rust saves the least because hyperfine's files are already
small and function-dense, so a whole-file baseline is not far from the chunks.

Recall and token-savings numbers are deterministic and depend only on the corpus
and the chunking, so the Ruby implementation gets **identical** figures on the
same pinned corpora (a cross-language cross-check); index and query latency are
engine-specific. Every query in SPEC-V2 §8 resolved to an existing file at the
pinned tags, so none were dropped.

> Reproduce a single language with, e.g.,
> `cce bench /path/to/sinatra --lang ruby --name "sinatra/sinatra@v4.1.1"`.
> The command writes this file (one language per invocation); the table above
> collates the four runs.
